# Investigate LinkedIn events scraping ‚Äî no public guest endpoint exists

Researched whether we can build a 'linkedin-company' or 'linkedin-event' source for our event importer suite using the same pattern as our LinkedIn jobs scraper (app/lib/job-search.server.ts:215 hitting linkedin.com/jobs-guest/jobs/api/seeMoreJobPostings/search).

## TL;DR

There is **no events-guest mirror** of the jobs-guest endpoint. LinkedIn deliberately keeps events behind the auth wall (events are a sign-up funnel; jobs are SEO bait). Every realistic endpoint pattern was probed and returns 302‚Üílogin or 404. Implementing a scraper requires either an authenticated li_at cookie (operationally heavy, ToS risk) or piggybacking on company websites that link out to LinkedIn events (already done for NetBenefit).

## Trigger

Asked whether we can build a LinkedIn-events analog of our LinkedIn jobs scraper, prompted by https://www.linkedin.com/events/netbenefitnetworking7470153274403422208/ which NetBenefit Software (company id 143) is hosting.

- `app/lib/job-search.server.ts:24-37` ‚Äî `searchIndeed` abuses Indeed's mobile GraphQL API with a hardcoded iOS app key (`indeed-api-key` = 161092c2017b5bbab13edb12461a62d5a833871e7cad6d9d475304573de67ac8). Pattern attributed to JobSpy.

## Prior art surveyed (external)

- JobSpy (https://github.com/speedyapply/JobSpy) ‚Äî only has /scrapers/linkedin/ for jobs. No event scraper.
- mhmdjaouhari/linkedin-company-events-scraper ‚Äî uses Puppeteer + li_at cookie.
- vertexcover-io/linkedin-spider ‚Äî uses li_at cookie or email/password, Playwright stealth.
- hansel-7/linkedin_stalker ‚Äî Playwright + li_at cookie.
- transitive-bullshit/linkedin-api ‚Äî Voyager API client, requires email/password login.
- Apify 'no-cookies' actors ‚Äî misleading marketing; they actually discover via Google site: queries and parse what OpenGraph leaks. Doesn't apply here since LinkedIn events aren't indexed.
- LinkedIn Marketing API (`/rest/events?q=organizer&organizer=urn:li:organization:{id}`) ‚Äî official, but requires the auth'd member to hold ADMINISTRATOR or CONTENT_ADMINISTRATOR role on the organization. Not viable for a third-party aggregator.

## Why this is by design

Jobs drive Google SERP traffic ‚Üí LinkedIn wants them indexed ‚Üí public guest endpoint.
Events drive sign-ups ‚Üí LinkedIn wants you to log in to see them ‚Üí no public endpoint.

## Viable paths if we ever want LinkedIn events

1. **Authenticated li_at cookie scraping**. Operationally heavy: cookie expires ~30 days, needs secret rotation, ToS risk, account risk. Pattern is well-documented (Voyager API: `/voyager/api/graphql?queryId=voyagerOrganizationDashEvents` with li_at and JSESSIONID cookies).
2. **Piggyback on company websites that embed their LinkedIn event URLs**. Already done for NetBenefit ‚Äî their /events page contains 3 LinkedIn event links we already extract via `netbenefit.server.ts`. Could generalize into a reusable 'company-events-page' importer with CSS selector config.
3. **Official Marketing API** ‚Äî requires admin role on each org. Not viable for our use case.
4. **Manual paste form** for one-off events ‚Äî admin pastes a LinkedIn event URL, manually fills title/date/location/description in a UI. No scraping; just structured manual entry.

## Recommendation

Don't build a generic LinkedIn-events importer. Either:
- Generalize the NetBenefit company-events-page pattern into a reusable importer (option 2 above) so other NL companies linking to their LinkedIn events from their own sites can be ingested, or
- Add a quick manual-paste form to the events suite (option 4) for the long tail.

Defer authenticated li_at scraping unless the volume justifies the operational pain.
