# Tiered MCP execute prompt + union CRUD for boring entities

## Why

The execute tool's prompt currently renders one verbose line per host
function (~150 chars each, 53 functions, ~7.5KB / ~1.9k tokens). Adding
the obvious next 15-20 functions (createPerson, createProduct,
createProject, createEducation, createTechnology, plus updateGroup,
updateNewsItem, deleteX variants, etc.) would push it to ~4k tokens.

Two distinct problems:
1. Prompt-rendering: O(N) verbose lines for every function.
2. Surface sprawl: bespoke createX/updateX/deleteX per entity is the
   wrong abstraction for entities that all share the same shape (name,
   description, website, visible, slug-generated).

## What

### Part 1: Tiered execute prompt rendering

Change describeEntries() in app/mcp/server.ts so the prompt renders
category-grouped name-only lists:

  Creation:
    createCompany, createGroup, createEvent, createJob, createEntity, ...
  Lookup:
    getCompanyByName, getGroupBySlug, getJobDetail, ...

Full signatures and descriptions stay co-located via host() and are
retrievable via the search tool (which already does this). The agent
learns: 'see a function name? call search(name) to get the signature.'

Prompt cost drops from ~1.9k tokens to ~400 tokens at current function
count, and stays sublinear with growth.

### Part 2: Union CRUD for boring entities

Add three new host functions:

- createEntity({ type, ...fields }) — type is one of:
    person | education | product | project | technology
  Each entity's required fields are defined inline in the dispatch
  switch; mismatched shapes fail-fast with a clear error.

- updateEntity({ type, id, ...fields }) — same union.

- deleteEntity({ type, id }) — same union.

These five entities all share the pattern: slug from name, visible
flag, hidden by default for review. They don't deserve 15 separate
host functions.

Bespoke functions are kept for entities with non-trivial shape:
events (date arrays, recurrence), jobs (company resolution,
isTechnical, workplaceType), news (article vs link vs source).

### Part 3: Add the lookups we need for the new surface

- getEntityBySlug(type, slug) — union lookup parallel to getCompanyByName.

## Acceptance

- buildExecuteDescription emits the tiered format. Total prompt drops
  to under ~500 tokens.
- createEntity/updateEntity/deleteEntity work for all 5 boring entities.
- getEntityBySlug returns { found, entity } shape matching
  getCompanyByName.
- search('createPerson') still returns full signature + description
  (covered by the existing host-function-docs branch in searchSpec).
- search('createEntity') returns the union docs.
- Build + lint + 527 tests pass.

## Out of scope (for now)

- Migrating events/jobs/news to a union. They keep their bespoke
  signatures because date/recurrence/workplaceType/company-resolve
  semantics make a one-shape-fits-all signature actively misleading.
- Adding createGroupSource / createEducationSource / etc. import
  sources for entities that don't have sources.
