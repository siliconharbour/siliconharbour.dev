# Job Importers Overview

This page documents how each importer works at a high level so we can quickly understand risk and maintenance cost.

## Core ATS Importers

| Importer    | How it works                                                                                              | Implementation style                                 | Reliability           |
| ----------- | --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- | --------------------- |
| Greenhouse  | Calls Greenhouse public JSON API (`boards-api.greenhouse.io`), maps jobs directly.                        | Clean API integration.                               | High                  |
| Lever       | Calls Lever public postings API (`api.lever.co`), builds description from structured sections.            | Clean API integration.                               | High                  |
| BambooHR    | Calls BambooHR JSON endpoints (`/careers/list` and per-job `/detail`).                                    | Clean API integration with detail fetch pass.        | High                  |
| Workable    | Calls Workable public JSON API (`/api/v1/widget/accounts/{slug}` for list, `/api/v2/.../jobs/{id}` for detail). | Clean API integration with detail fetch pass.  | High                  |
| Jibe        | Calls Jibe JSON API (`/api/jobs?limit=100&page=N`). Returns full descriptions in one call, paginated.     | Clean API integration. Single-step fetch.            | High                  |
| ADP         | Parses ADP WorkforceNow recruitment pages, extracts job data from embedded JSON structures.               | Structured data extraction from HTML.                | Medium-High           |
| CareerBeacon| Fetches individual job pages, extracts JSON-LD `JobPosting` schema. Supports direct URLs or listing scrape.| JSON-LD extraction. Multi-URL with p-limit(5).      | Medium-High           |
| iCIMS       | Two-step: scrapes listing HTML for job URLs, then extracts JSON-LD `JobPosting` from detail pages.        | HTML listing scrape + JSON-LD detail.                | Medium-High           |
| Workday     | Uses Workday JSON endpoints (`/wday/cxs/...`) with browser-like user agent and custom identifier parsing. | API-based, but has platform quirks.                  | Medium-High           |
| Ashby       | Scrapes HTML page, extracts embedded `window.__appData`, then fetches each job detail page similarly.     | Structured data extraction from HTML shell.          | Medium                |
| Rippling    | Scrapes HTML page, extracts embedded `__NEXT_DATA__`, then fetches each job detail page.                  | Structured data extraction from framework internals. | Medium                |
| Collage     | Scrapes rendered HTML and parses listing/detail content using linkedom DOM parsing.                        | Direct HTML parsing.                                 | Medium-Low            |
| Custom      | Dispatches to per-company custom scrapers. Some use APIs; many parse bespoke HTML.                        | Mixed, often site-specific.                          | Mixed (Medium to Low) |

## Notes By Importer

### Greenhouse

- Source: `app/lib/job-importers/greenhouse.server.ts`
- Approach: Public API with optional `content=true` for descriptions.
- Assessment: Clean and straightforward. Lowest maintenance burden.

### Lever

- Source: `app/lib/job-importers/lever.server.ts`
- Approach: Public API for listing and optional single-job detail.
- Assessment: Clean and stable in practice.

### BambooHR

- Source: `app/lib/job-importers/bamboohr.server.ts`
- Approach: JSON listing + per-job JSON detail fetch.
- Assessment: Clean; detail fetched one-by-one. Listing `externalId` preserved through detail merge to avoid URL-parsing issues.

### Workable

- Source: `app/lib/job-importers/workable.server.ts`
- Approach: Public JSON API at `apply.workable.com`. V1 widget endpoint for listing (returns all jobs, no pagination), V2 endpoint for per-job description/requirements/benefits. Uses p-limit(5) for concurrent detail fetches.
- Identifier: Company slug from `apply.workable.com/{slug}`.
- Assessment: Very clean. Public API, no auth, highly reusable across any Workable customer.

### Jibe

- Source: `app/lib/job-importers/jibe.server.ts`
- Approach: Public JSON API at `{domain}/api/jobs?limit=100&page=N`. Returns everything (title, full HTML description, location with coordinates, department, categories) in one call. Pagination via page number.
- Identifier: Full domain (e.g., `jobs.symphonyai.com`). Each Jibe customer has their own domain.
- Quirk: API wraps each job in a `{ data: {...} }` envelope.
- Assessment: Very clean. Richest data of any importer (includes salary fields, geo coordinates). No detail fetch needed.

### ADP

- Source: `app/lib/job-importers/adp.server.ts`
- Approach: Fetches ADP WorkforceNow recruitment pages, extracts job data from embedded JSON structures in the HTML.
- Identifier: Company client ID (`cid:ccId`) or full ADP recruitment URL.
- Assessment: Works but ADP page structure can vary between tenants.

### CareerBeacon

- Source: `app/lib/job-importers/careerbeacon.server.ts`
- Approach: Two input modes — direct job URL(s) or company ID/slug for listing page scrape. Extracts JSON-LD `JobPosting` schema from individual job pages. Uses p-limit(5) for concurrent fetches.
- Identifier: Job URL(s), company ID, or employer slug.
- Quirk: Listing pages may be blocked by Cloudflare; direct job URLs are recommended.
- Assessment: JSON-LD extraction is reliable. Listing scrape is best-effort.

### iCIMS

- Source: `app/lib/job-importers/icims.server.ts`
- Approach: Two-step. Fetches listing page (`?in_iframe=1`) and parses job URLs from HTML anchor tags. Then fetches each detail page (also with `?in_iframe=1`) and extracts JSON-LD `JobPosting` schema. Uses p-limit(5) for concurrent detail fetches.
- Identifier: Subdomain prefix (e.g., `uscareers-repairify` from `uscareers-repairify.icims.com`).
- Quirk: JSON-LD only present on the `?in_iframe=1` version of pages.
- Assessment: Reliable JSON-LD extraction. Standard iCIMS platform behavior across customers.

### Workday

- Source: `app/lib/job-importers/workday.server.ts`
- Approach: JSON API via `POST` to Workday `cxs` endpoints, then detail requests per posting.
- Quirks: Cloudflare protection on some Workday tenants can block automation. Source identifier encoding is custom (`company:site[:searchText]`). External ID regex broadened to handle lowercase/underscore variants.
- Fallback: May need manual workflow for heavily protected tenants.
- Assessment: Functionally API-oriented, but operationally fragile because anti-bot behavior is outside our control.

### Ashby

- Source: `app/lib/job-importers/ashby.server.ts`
- Approach: Reads `window.__appData` out of HTML for both list and details.
- Assessment: Not as fragile as blind HTML scraping, but still coupled to page bootstrap internals.

### Rippling

- Source: `app/lib/job-importers/rippling.server.ts`
- Approach: Reads `__NEXT_DATA__` from Next.js pages and extracts dehydrated query payloads.
- Assessment: Similar to Ashby: structured, but tied to frontend serialization details.

### Collage

- Source: `app/lib/job-importers/collage.server.ts`
- Approach: DOM parsing of rendered HTML on listing and detail pages using linkedom.
- Assessment: More brittle than API-based importers; likely to need updates when templates change.

### Custom Dispatcher

- Source: `app/lib/job-importers/custom.server.ts`
- Approach: Chooses a company-specific scraper by `sourceIdentifier`.
- Assessment: Useful escape hatch, but inherently maintenance-heavy.

## Practical Summary

- **Cleanest overall**: Greenhouse, Lever, BambooHR, Workable, Jibe.
- **JSON-LD based** (reliable structured data): CareerBeacon, iCIMS.
- **Most operationally quirky but still API-oriented**: Workday, ADP.
- **Most likely to break on frontend changes**: Collage and many `custom` scrapers.
- **Best long-term strategy**: Prefer provider APIs where available and reserve `custom` for unavoidable cases. When a company moves to a supported ATS, migrate off the custom scraper (as done with Enaimco: custom -> BambooHR).
